- [x] CHK003 - Are PHP-FPM pool configuration requirements documented (workers, listen socket, timeout)? [Completeness, Gap]
  → **GAP**: Not in spec. Acceptable for plan-level, but at minimum listen socket type (TCP vs Unix) should be specified.
- [x] CHK004 - Are Docker internal network requirements defined (service name resolution, isolation from host network)? [Completeness, Gap]
  → **GAP**: No network requirements. Impacts PostgreSQL security (CHK041).
- [x] CHK005 - Is the Nginx container image version specified to match a known-good or production-aligned version? [Completeness, Gap]
  → **GAP**: Assumption says "Nginx (not Apache)" but no version pinned.
- [x] CHK006 - Are Composer dependency management requirements specified (cache mount, vendor mount, or copy strategy)? [Completeness, Gap]
  → **PARTIAL**: FR-006 states presence only. Mount/copy strategy for `vendor/` left to plan.
- [x] CHK007 - Is a `docker-compose.override.yaml` strategy addressed for per-developer customizations without modifying the committed file? [Completeness, Gap]
  → **GAP**: Not mentioned.
- [x] CHK008 - Are PHP opcache configuration requirements for local dev mode specified (e.g., revalidation frequency)? [Completeness, Gap]
  → **GAP**: FR-005 lists `opcache` as required extension but no dev-mode config (revalidation = 0 vs default behavior).
- [x] CHK009 - Are `.gitignore` update requirements documented for files generated by the Docker setup (e.g., `.env.local`, override files)? [Completeness, Gap]
  → **GAP**: FR-007 says `.env.local` not committed but no `.gitignore` requirement stated.
- [x] CHK010 - Are requirements for the README beyond the OAuth2 registration section (FR-014) defined? [Completeness, Spec §FR-014]
  → **GAP**: SC-003 requires a developer to set up the environment using "only the README" — but README content requirements are not defined beyond FR-014's OAuth2 section.

## Requirement Clarity

- [x] CHK011 - Is "within 2 minutes" (US-1 AC-1) measured from which point: post-image-pull, or including cold image download? [Clarity, Spec §US-1]
  → **GAP**: Ambiguous. Cold image pull for 4 containers can easily exceed 2 minutes alone.
- [x] CHK012 - Is "all containers start without errors" (US-1 AC-1) defined with a concrete observable signal (health check passing, exit code, log line)? [Clarity, Spec §US-1]
  → **GAP**: "without errors" is subjective. No concrete signal defined (e.g., `docker-compose ps` shows all healthy).
- [x] CHK013 - Is "clear error message" (US-1 AC-3) specified with required format, output stream (stdout/stderr), and content expectations? [Ambiguity, Spec §US-1]
  → **GAP**: "a clear error message indicates which required variable is missing" — no format, stream, or exact content requirement.
- [x] CHK014 - Is "within 5 minutes" (SC-001) measured from `git clone` completion or from the first `docker-compose up -d` invocation? [Clarity, Spec §SC-001]
  → **GAP**: SC-001 says "within 5 minutes of cloning" — ambiguous whether image pull time counts.
- [x] CHK015 - Is "no unresolved peer dependency errors" (US-3 AC-1) clarified to distinguish errors from warnings? [Clarity, Spec §US-3]
  → **GAP**: `npm install` with Bootstrap 5 + Webpack Encore typically produces peer dependency warnings. Spec does not distinguish.
- [x] CHK016 - Is "compiled assets are generated in the `public/` directory" (US-3 AC-2) specified with exact subdirectory path and expected output file types? [Clarity, Spec §US-3]
  → **GAP**: Webpack Encore outputs to `public/build/` — spec says "public/" which is imprecise. Must be `public/build/` for the Nginx root to serve correctly.
- [x] CHK017 - Is "identical to the Platform.sh build hook" (SC-005) defined with measurable comparison criteria (hash, file list, byte size)? [Clarity, Measurability, Spec §SC-005]
  → **GAP + INCONSISTENCY**: SC-005 references `npm ci && npm run build`. US-3 AC-1 says `npm install`. Platform.sh uses `npm ci` (locked install); spec uses `npm install` (resolves latest). Output may differ. Need to reconcile.
- [x] CHK018 - Is the `npm run watch` behavior (US-3 AC-3) specified beyond "recompiles on each save" (e.g., latency, scope of watched files)? [Clarity, Spec §US-3]
  → **ACCEPTABLE**: `encore dev --watch` behavior is well-defined by Webpack Encore. Spec is adequate at this level.
- [x] CHK019 - Is `XDEBUG_MODE=debug` the only required change to enable Xdebug, or are additional IDE/port configuration steps required? [Clarity, Spec §FR-005b]
  → **GAP**: FR-005b says "enable via `.env.local`" without rebuilding — but Xdebug 3 also requires `XDEBUG_CLIENT_HOST` and port config for IDE connection. IDE integration scope is unstated.

## Requirement Consistency

- [x] CHK020 - Do version requirements (PHP 8.3, PostgreSQL 16, Node 20 LTS) align with the versions currently declared in `.platform.app.yaml`? [Consistency, Spec §FR-002/003/004]
  → **PARTIAL PASS**: PHP 8.3 ✓ (`.platform.app.yaml`: `type: php:8.3`). Node 20 ✓ (`nvm use 20`). PostgreSQL version not in `.platform.app.yaml` — only `"database:postgresql"` with no version pin. Cannot fully verify FR-003.
- [x] CHK021 - Are the `var/` and `public/` mount paths referenced in the assumptions consistent with the bind-mount paths that `docker-compose.yaml` will define? [Consistency, Spec §Assumptions]
  → **CRITICAL DEFECT**: `.platform.app.yaml` only mounts `/var` — `public/` is NOT a mount. Spec assumption says "var/ AND public/ mount paths defined in `.platform.app.yaml` are mirrored." This is factually wrong. `public/` is the Nginx document root and Encore build output directory, not a Platform.sh mount. Assumption must be corrected.
- [x] CHK022 - Does the Xdebug requirement (FR-005b) align with a specified IDE or debugger client? Or is IDE integration intentionally out of scope? [Consistency, Gap, Spec §FR-005b]
  → **NEEDS CLARIFICATION**: IDE integration is not mentioned. Either explicitly declare it out of scope or specify minimum Xdebug 3 remote config requirements.
- [x] CHK023 - Is the named volume requirement (FR-008) consistent with `docker-compose down -v` destroying data in US-4 AC-2, such that the volume name is unambiguous? [Consistency, Spec §FR-008, §US-4]
  → **PASS**: FR-008 and US-4 AC-2 are consistent. Named volume persists across down/up; `-v` flag removes it.
- [x] CHK024 - Are the OAuth2 provider placeholder variables in FR-014 consistent with the provider list defined in spec 002-user-auth-oauth2? [Consistency, Spec §FR-014]
  → **INCONSISTENCY**: FR-014 says "Google, GitHub" — spec 002 covers only Google OAuth2. No GitHub provider appears in spec 002. Either spec 002 is incomplete or FR-014 is over-specified.
- [x] CHK025 - Does FR-012 (no modification of Platform.sh files) conflict with the assumption that `var/` and `public/` mounts mirror `.platform.app.yaml` paths? [Conflict, Spec §FR-012, §Assumptions]
  → **CONFLICT**: FR-012 says docker config "MUST NOT modify or reference Platform.sh configuration files." The assumption says mount paths are "mirrored" from `.platform.app.yaml` — implementers must read that file to know the paths. These requirements conflict. Additionally compounded by CHK021: `public/` is not a Platform.sh mount, so the assumption is doubly incorrect.

## Scenario Coverage

- [x] CHK031 - Are requirements defined for the case where a Docker image pull fails mid-setup (network interruption or registry unavailability)? [Coverage, Edge Case, Gap]
  → **GAP**: Edge cases section covers port conflicts and missing `.env.local` only. Image pull failure not addressed.
- [x] CHK032 - Is the behavior specified when a developer runs `npm run build` before `npm install` (empty node_modules)? [Coverage, Edge Case, Spec §US-3]
  → **GAP**: US-3 defines the happy path only. Running `build` before `install` produces an opaque error.
- [x] CHK033 - Are host OS compatibility requirements defined (Windows WSL2, macOS, Linux) given UID/GID handling differs per platform? [Coverage, Gap]
  → **GAP**: `${UID:-1000}:${GID:-1000}` is unset on Windows by default. WSL2 handling differs. Spec doesn't scope supported host OSes.
- [x] CHK034 - Is the behavior defined when `.env.local` exists but contains syntactically valid yet semantically wrong values (e.g., incorrect DB password)? [Coverage, Edge Case]
  → **GAP**: US-1 AC-3 covers missing `.env.local` / missing variables only. Wrong credentials = silent connection failure at runtime.
- [x] CHK035 - Is the recovery flow defined if a developer runs `docker-compose up` before running `composer install` (missing vendor/)? [Coverage, Gap]
  → **GAP**: Not addressed. PHP container will start but Symfony app will fail with an opaque autoload error.

## Edge Case Coverage

- [x] CHK036 - Is the port conflict scenario scoped only to 8000 and 5432, or does it also cover any ports exposed by the Node.js container? [Edge Case, Spec §Edge Cases]
  → **GAP**: `package.json` has `dev-server` script (`encore dev-server`) which runs on port 8080. Not mentioned in port conflict handling.
- [x] CHK037 - Is the behavior specified when `UID` or `GID` environment variables are unset on the host at `docker-compose up` time? [Edge Case, Spec §FR-005a]
  → **PARTIAL PASS**: `${UID:-1000}:${GID:-1000}` handles unset with default 1000. Not documented as intentional behavior — implementers may change it unknowingly.
- [x] CHK038 - Is the expected behavior defined when the PHP-FPM container attempts a database connection before PostgreSQL initialization is complete? [Edge Case, Gap]
  → **GAP**: No startup ordering requirement. `restart: unless-stopped` provides a workaround but not a defined requirement. Health check dependency (CHK001) would resolve this.
- [x] CHK039 - Is the behavior defined when `docker-compose down -v` is executed while a migration is actively running? [Edge Case, Gap]
  → **GAP**: Not addressed. Low-probability but destructive.
- [x] CHK040 - Is running `docker-compose up` without internet access (all images already pulled) addressed? [Edge Case, Gap]
  → **PASS / NOT APPLICABLE**: Standard Docker behavior (cached images). No spec-level requirement needed.
