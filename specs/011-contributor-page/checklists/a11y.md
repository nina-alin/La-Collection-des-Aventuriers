# Accessibility Requirements Quality Checklist: Page Contributeur — Suggestions

**Purpose**: Pre-planning gate — validate WCAG 2.1 AA requirement completeness, clarity, and measurability before implementation planning
**Created**: 2026-05-30
**Feature**: [spec.md](../spec.md)

## Requirement Completeness

- [x] CHK001 - Are ARIA label requirements specified field-by-field, not just "all fields" globally? [Completeness, Spec §FR-025] → FR-025 covers globally; FR-058–064 add component-level specifics
- [x] CHK002 - Are `aria-live` region requirements defined for: validation errors, toast notifications, polling updates ("Mon suivi"), and diff count changes? [Completeness, Spec §FR-025] → FR-058
- [x] CHK003 - Is focus management defined for each transition: step forward, step backward, error appearance, modal/dropdown open, modal/dropdown close? [Completeness, Spec §FR-025] → FR-059
- [x] CHK004 - Are keyboard alternative requirements defined for the drag-and-drop upload zone (file selection without dragging)? [Completeness, Gap] → FR-035, FR-061
- [x] CHK005 - Are screen reader announcement requirements defined for diff changes at step 4 (e.g., "3 champs modifiés" update)? [Completeness, Gap] → FR-058 (aria-live="polite" on status messages)
- [x] CHK006 - Is focus order defined for autocomplete dropdown interactions (open, navigate items, select, close without selecting)? [Completeness, Gap] → FR-032, FR-061
- [x] CHK007 - Are focus management requirements defined when the quota warning (FR-018) appears or disappears? [Completeness, Gap] → FR-059 (quota warning triggers as blocking error → focus to first error field)
- [x] CHK008 - Are requirements defined for accessible labeling of the stepper (step progress, current step, completed steps)? [Completeness, Gap] → FR-060
- [x] CHK009 - Are requirements defined for error summary announcement when multiple fields fail simultaneously? [Completeness, Gap] → FR-058 (individual field errors via aria-live="assertive"; no global summary required)

## Requirement Clarity

- [x] CHK010 - Is "label ARIA explicite" defined — does this require `aria-label`, `aria-labelledby`, or a visible `<label for>` element? [Clarity, Spec §FR-025] → FR-025 requires compliance with WCAG 2.1 AA 1.3.1; implementation chooses appropriate technique
- [x] CHK011 - Is the keyboard navigation list in FR-025 (Tab, Shift+Tab, Enter, Espace) exhaustive — are Arrow keys required for stepper, radio buttons, or autocomplete? [Clarity, Spec §FR-025] → FR-032 adds Arrow keys for autocomplete dropdown; radio buttons use native browser behavior
- [x] CHK012 - Is "sans erreur critique" in SC-007 bound to a specific axe-core rule profile and violation severity level (critical vs serious)? [Clarity, Spec §SC-007] → SC-007 updated: axe-core, critical + serious violations
- [x] CHK013 - Is "focus DOIT être géré explicitement à chaque changement d'étape" defined — which element receives focus (step heading, first field, error message)? [Clarity, Spec §FR-025] → FR-059
- [x] CHK014 - Are color contrast requirements specified beyond "AA" for interactive states — hover rings, focus outlines, disabled states? [Clarity, Gap] → WCAG 2.1 AA includes 1.4.11 Non-text Contrast (3:1 for UI components); covered by FR-025 AA compliance
- [x] CHK015 - Is `aria-live="assertive"` applied uniformly to all validation errors, or are some errors `polite`? [Clarity, Spec §FR-025] → FR-058

## Requirement Consistency

- [x] CHK016 - Do WCAG keyboard navigation requirements (FR-025) apply to the mobile tab switcher (FR-024) as well as the desktop wizard? [Consistency, Spec §FR-024, FR-025] → FR-025 applies to full page; FR-060 covers stepper on all viewports
- [x] CHK017 - Are `aria-live` requirements consistent between real-time validation messages and polling update notifications? [Consistency, Spec §FR-007, FR-021, FR-025] → FR-058
- [x] CHK018 - Is keyboard navigation required for the drag-and-drop zone consistent with the global keyboard navigation requirement in FR-025? [Consistency, Spec §FR-013, FR-025] → FR-035 + FR-061 (file input button always present as keyboard alternative)
- [x] CHK019 - Are accessibility requirements applied to the "Mon suivi" panel and its filter buttons, not only the wizard? [Consistency, Spec §FR-021, FR-025] → FR-025 applies to full page; FR-064 covers dashboard region

## Scenario Coverage

- [x] CHK021 - Are requirements defined for screen reader behavior during the is-checking (spinner) validation state — is the spinner announced? [Coverage, Spec §FR-007] → FR-061 (role="status", aria-label="Vérification en cours...")
- [x] CHK022 - Are accessibility requirements defined for the refusal detail expansion triggered by the FR-023 info button (expand/collapse pattern)? [Coverage, Spec §FR-023] → FR-062
- [x] CHK023 - Are accessible error recovery requirements defined after a failed file upload (focus return, error announcement)? [Coverage, Edge Case] → FR-063
- [x] CHK024 - Are accessibility requirements defined for the rank/progress bandeau (FR-001–002) — is progression data announced to screen readers? [Coverage, Spec §FR-001, FR-002] → FR-064
